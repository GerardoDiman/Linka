## Detalle de Correcciones Aplicadas

### 1. CORS — Rechazo de orígenes no autorizados
**Archivo:** `supabase/functions/_shared/cors.ts`

**Antes:** Cuando un request llegaba de un origen no incluido en `ALLOWED_ORIGINS`, se devolvían los headers de `linka-studio.com`. Aunque el navegador rechazaba la respuesta, el request sí se ejecutaba en el servidor.

**Después:** Los orígenes no autorizados reciben headers vacíos. El servidor ya no valida requests de dominios arbitrarios.

```typescript
// Antes
const allowed = ALLOWED_ORIGINS.includes(origin) ? origin : ALLOWED_ORIGINS[0]

// Después
if (!ALLOWED_ORIGINS.includes(origin)) {
    return { 'Access-Control-Allow-Origin': '', ... }
}
return { 'Access-Control-Allow-Origin': origin, ... }
```

### 2. RLS — Perfiles restringidos al propio usuario
**Archivo:** `supabase/fix_sync_profiles.sql`

**Antes:** `USING (true)` — cualquier usuario autenticado podía hacer `SELECT * FROM profiles` y ver emails, roles y planes de TODOS los usuarios.

**Después:** `USING (auth.uid() = id)` — cada usuario solo ve su propio perfil.

**⚠️ ACCIÓN MANUAL REQUERIDA** — ver sección "Acciones Pendientes" más abajo.

### 3. n8n-bridge — Verificación de firma para webhooks
**Archivo:** `supabase/functions/n8n-bridge/index.ts`

**Antes:** Las rutas de Auth Hooks y Database Webhooks saltaban completamente la autenticación. Un atacante podía enviar payloads simulando hooks y triggear emails arbitrarios vía n8n.

**Después:** Se valida un `x-webhook-signature` header contra `SUPABASE_WEBHOOK_SECRET` antes de procesar cualquier webhook.

**⚠️ ACCIÓN MANUAL REQUERIDA** — ver sección "Acciones Pendientes" más abajo.

### 4. Checkout Session — Prevención de Open Redirect
**Archivo:** `supabase/functions/create-checkout-session/index.ts`

**Antes:** `success_url` y `cancel_url` se construían con `req.headers.get("origin")`, permitiendo que un atacante redirigiera post-pago a un sitio malicioso.

**Después:** Siempre usa `SITE_URL` configurado en el servidor. El header `Origin` del request se ignora.

```typescript
// Antes
const origin = req.headers.get("origin") || SITE_URL

// Después
const origin = SITE_URL
```

### 5. Promise.race — Cleanup de timers con withTimeout()
**Archivos:**
- `src/lib/withTimeout.ts` (nuevo)
- `src/context/AuthContext.tsx`
- `src/lib/cloudSync.ts`

**Antes:** `Promise.race` con `setTimeout` sin limpiar el timer cuando la promise principal ganaba. Dejaba timers colgados y la query perdedora seguía consumiendo memoria.

**Después:** Utility `withTimeout()` que acepta `PromiseLike<T>` (compatible con Supabase's PostgrestBuilder) y limpia el timer en `finally`.

```typescript
export async function withTimeout<T>(
    thenable: PromiseLike<T>, ms: number, label = 'Operation'
): Promise<T> {
    let timeoutId: ReturnType<typeof setTimeout>
    const timeout = new Promise<never>((_, reject) => {
        timeoutId = setTimeout(() => reject(new Error(`${label} timed out after ${ms}ms`)), ms)
    })
    try {
        return await Promise.race([Promise.resolve(thenable), timeout])
    } finally {
        clearTimeout(timeoutId!)
    }
}
```

Reemplazó 4 instancias de `Promise.race` manuales:
- `AuthContext.tsx` — profile fetch (3s timeout)
- `cloudSync.ts` — session refresh (3s), upsert (4s), data fetch (4s)

### 6. Errores silenciosos — Logging real en catch blocks
**Archivos:**
- `src/context/AuthContext.tsx`
- `src/hooks/useCloudSync.ts`

**Antes:** Bloques `catch {}` vacíos o `catch { /* vacío */ }` que tragaban errores sin notificar al usuario ni dejar rastro.

**Después:** Todos los `catch` usan `logger.error()` con contexto. El `signOut()` ya no usa `finally` para el redirect, evitando navegación si `signOut()` falla.

---

### 7. Fetch duplicado de plan_type eliminado
**Archivo:** `src/hooks/useCloudSync.ts`

**Antes:** `plan_type` se obtenía de Supabase en 3 lugares (AuthContext, useCloudSync, DashboardPage), generando hasta 3 queries redundantes por carga de página.

**Después:** Se eliminó el `useEffect` de `fetchPlan` en `useCloudSync.ts`. `AuthContext` es la única fuente de verdad para `plan_type`.

---

### 8. eslint-disable resueltos con refs
**Archivos:**
- `src/hooks/useCloudSync.ts`
- `src/hooks/useGraphFilters.ts`

**Antes:** 3 `eslint-disable-line react-hooks/exhaustive-deps` ocultaban dependencias faltantes, causando stale closures donde la sincronización podía usar colores o filtros desactualizados.

**Después:**
- `useCloudSync.ts`: Se usan `useRef` para `handleSync`, `setSelectedPropertyTypes`, `setHiddenDbIds`, `setHideIsolated` y `setCustomColors`. Los refs se actualizan en effects separados, y el effect principal depende solo de `session`.
- `useGraphFilters.ts`: Se agregó `initialColorsLoadedRef` como guard para evitar re-ejecución infinita, permitiendo incluir todas las dependencias en el array.

### 10. PII redactada en logs
**Archivos:**
- `supabase/functions/notion-sync/index.ts`
- `supabase/functions/n8n-bridge/index.ts`
- `supabase/functions/stripe-webhook/index.ts`

**Después:** Todos los logs usan `user.id.substring(0, 8) + '...'` para ofuscar IDs. Emails eliminados de logs.

### 11. Stripe webhook — Idempotencia agregada
**Archivo:** `supabase/functions/stripe-webhook/index.ts`

**Antes:** `checkout.session.completed` se procesaba sin verificar si ya había sido procesado. Stripe puede reenviar webhooks hasta 3 veces.

**Después:** Se verifica si `plan_type` ya es `'pro'` y `stripe_customer_id` coincide antes de actualizar. Si ya fue procesado, responde `{ received: true, skipped: true }` con status 200.

---

### 12. signOut sin hard reload en finally
**Archivo:** `src/context/AuthContext.tsx`

**Antes:** `window.location.href = '/'` estaba en el bloque `finally`, ejecutándose incluso si `signOut()` fallaba.

**Después:** El redirect se ejecuta solo después del bloque try/catch, no dentro de `finally`.

### 14. Dynamic imports para ExportButton
**Archivo:** `src/components/dashboard/ExportButton.tsx`

**Antes:** `html-to-image` (118KB) y `downloadjs` (5KB) se importaban estáticamente, inflando el bundle principal.

**Después:** Ambas dependencias usan `import()` dinámico dentro de los handlers. Verificado en build: `download-Q6NEzodW.js` aparece como chunk separado.

```typescript
const handleExport = async () => {
    const { toBlob } = await import("html-to-image")
    const { default: download } = await import("downloadjs")
    // ...
}
```

## Acciones Pendientes (Manual)

### ⚠️ OBLIGATORIA 1: Aplicar política RLS en Supabase Dashboard

La política actual `"Public profiles are viewable by everyone"` con `USING (true)` expone todos los perfiles. Ejecutar en el **SQL Editor de Supabase**:

```sql
-- Paso 1: Eliminar la política existente
DROP POLICY IF EXISTS "Public profiles are viewable by everyone" ON public.profiles;

-- Paso 2: Crear la nueva política restrictiva
CREATE POLICY "Users can view own profile"
ON public.profiles FOR SELECT
USING (auth.uid() = id);
```

**Verificación:** Después de aplicar, intentar ejecutar `SELECT * FROM profiles` como un usuario autenticado. Solo debería devolver 1 fila (la del propio usuario).

> **Nota:** Si alguna parte de la app necesita acceder a datos de otros usuarios (ej: un panel admin), crear una política adicional con `USING (auth.uid() IN (SELECT id FROM profiles WHERE role = 'admin'))` o usar el service role key desde una edge function.

---

### ⚠️ OBLIGATORIA 2: Configurar SUPABASE_WEBHOOK_SECRET

El n8n-bridge ahora requiere un secret compartido para autenticar webhooks de Supabase.

1. **Generar un secret seguro:**
   ```bash
   openssl rand -hex 32
   ```

2. **Configurar en Supabase Edge Functions:**
   - Dashboard → Edge Functions → Settings (o via CLI)
   ```bash
   supabase secrets set SUPABASE_WEBHOOK_SECRET=<tu-secret-generado>
   ```

3. **Configurar en Supabase Webhooks:**
   - Dashboard → Database → Webhooks → Editar cada webhook que apunte a n8n-bridge
   - Agregar header: `x-webhook-signature: <tu-secret-generado>`

4. **Configurar en Auth Hooks (si aplica):**
   - Dashboard → Authentication → Hooks → Editar hooks que llamen a n8n-bridge
   - Agregar el mismo header `x-webhook-signature`

**Verificación:** Disparar un evento que active el webhook y verificar en los logs de Edge Functions que aparezca `"Webhook signature verified successfully"`.

## Principios Aplicados

- **OWASP A01:2021 (Broken Access Control):** RLS restrictiva, CORS estricto, open redirect eliminado
- **OWASP A04:2021 (Insecure Design):** Webhook authentication, idempotencia en pagos
- **OWASP A09:2021 (Security Logging Failures):** PII redactada, errores logueados correctamente
- **DRY:** Eliminada triple fuente de `plan_type`, utility `withTimeout` reutilizable
- **Resource Cleanup:** Todos los timers se limpian en `finally`
- **Fail-Fast:** Errores se propagan y loguean en lugar de tragarse silenciosamente
